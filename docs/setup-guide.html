1. What you're building

A private cluster of ten NixOS machines that configure themselves. You point a fresh machine at a Git repository, give it one secret (a 1Password token), and it does the rest: fetches its credentials, joins your private network, mounts shared storage, joins the container cluster, and reports in to a management UI.

YOU (admin workstation, on the tailnet) │ git push config ▲ manage via browser ▼ │ GitHub ──────────────► Portainer UI (https://node01:9443) │ nixos-rebuild ▲ ▼ │ agent on every node ┌──────────────────────────────────────────────────────────┐ │ node01 ─ node03: Swarm managers node04 ─ node10: workers │ every node: Tailscale ▪ Docker ▪ Ceph client ▪ Vector │ │ private mesh network (Tailscale WireGuard, 100.x.y.z) │ └──────────────────────────────────────────────────────────┘ │ fetch secrets at boot │ ship logs ▼ ▼ 1Password (op://Infrastructure/…) Better Stack (alerts → webhooks)

The three rules that make this safe

  1. Git is the source of truth, and it holds zero secrets. The public repo is the framework anyone can reuse; your private ops repo holds your real inventory. Neither ever contains a password, key, or token.
  2. One secret per machine. The only credential you ever place on a node by hand is a 1Password service account token. Everything else (Tailscale keys, Ceph keys, Swarm tokens, logging tokens) is fetched from 1Password at boot, by reference.
  3. Nothing listens on the public internet. All management traffic flows over the Tailscale mesh. The firewall trusts only the tailnet interface.

3. Step 1 — Set up 1Password

3.1 Create the vaults

Create two vaults (1Password app → New Vault):

VaultPurposeWho can read it
InfrastructureRuntime secrets the nodes fetch at boot. Item names below are read by the code exactly as written — no spaces, match the capitalization.You + the cluster's service account (read-only)
AdminMaster copies and break-glass credentials: the service-account token itself, account logins, recovery codes. Never grant any service account access to this vault.You only
Why two vaults? If a node is ever compromised, the attacker gets the node's service-account token — which can read only the Infrastructure vault. Your account logins, recovery codes, and the token master copies stay out of reach, and you revoke one service account to lock everything down.

3.3 Create the service account

  1. Go to my.1password.com → Developer → Service Accounts → Create (docs: developer.1password.com/docs/service-accounts).
  2. Name it after the environment, e.g. nixos-cluster-prod.
  3. Grant Read access to the Infrastructure vault. Nothing else. Not the Admin vault.
  4. The token (ops_…) is shown once. Immediately save it in the Admin vault as item Service Account Token — Production.

4. Step 2 — Set up Tailscale

  1. Create a tailnet at login.tailscale.com and install Tailscale on your admin workstation (your laptop must be on the tailnet to manage the cluster).
  2. Check DNS → MagicDNS is enabled (it is by default) — nodes find each other by hostname (node01) instead of IP.
  3. Generate the auth key (Settings → Keys → Generate auth key): Reusable ✓, Pre-approved ✓ (so nodes don't wait for manual device approval), expiry 90 days.
  4. Store it in 1Password: Infrastructure → Tailscale → credential.
Key expiry tip. The auth key only matters when a node joins. Existing nodes stay connected after it expires — you only need a fresh key (same 1Password item, updated) when adding or rebuilding nodes.

5. Step 3 — Set up the Git repositories

Two repositories, deliberately split:

RepoVisibilityContains
NixOS (the framework — fork github.com/SEBK4C/NixOS)PublicShared modules, specs, this guide. Generic and secret-free; safe to publish, easy to share.
NixOS-Ops (your inventory — copy the layout from the framework's README)PrivateYour real settings, per-node hardware configs, host-specific software, workload stacks, and the support-agent issue trail. Private because operational data (logs, addresses, incident details) shouldn't be public, even though it also contains no secrets.
  1. Fork the framework repo (or use it directly if you're me).
  2. Create your private ops repo with: flake.nix (composes the framework's nixosModules.cluster), cluster-settings.nix, and a hosts/nodeXX/ directory per node — see github.com/SEBK4C/NixOS-Ops layout in the framework README.
  3. Edit cluster-settings.nix in the ops repo: your SSH public key in adminSSHKeys, Ceph monitor addresses (or leave empty to skip Ceph), opsFlake pointing at your ops repo.
  4. Create the read-only GitHub PAT (Section 3.2, item GitHub) — nodes use it to fetch the private repo. Use a fine-grained token scoped to only the ops repo with Contents: Read.
  5. On GitHub, enable passkeys/2FA and branch protection on main of both repos. Whoever controls these repos controls your machines.

6. Step 4 — Bring up the first node

  1. Install NixOS minimal on the machine (nixos.org/download → minimal ISO). Any standard install works; give it disk and network.
  2. Capture the hardware config — on the node: 
    nixos-generate-config --show-hardware-config > hardware-configuration.nix
    Copy this file into your ops repo at hosts/node01/hardware-configuration.nix, commit, push. (This is the only per-node file; repeat for each node.)
  3. Place the one manual secret — on the node, as root: 
    mkdir -p /etc/op
echo 'ops_YOUR_SERVICE_ACCOUNT_TOKEN' > /etc/op/token
chmod 600 /etc/op/token
  4. Build — the first build needs the GitHub token inline (1Password isn't wired up yet): 
    export NIX_CONFIG="experimental-features = nix-command flakes
access-tokens = github.com=github_pat_YOUR_READONLY_TOKEN"
nixos-rebuild switch --flake github:YOUR_USER/NixOS-Ops#node01
  5. Watch it bootstrap: validates 1Password → joins the tailnet → (mounts Ceph) → initializes the Swarm. Check progress with journalctl -u op-bootstrap -u tailscale-join -u swarm-join -f.
From now on, rebuilds are one command. Every node gets a cluster-rebuild helper that fetches the GitHub token from 1Password per run — nothing stored on disk. cluster-rebuild (apply), cluster-rebuild test (try without persisting), cluster-rebuild boot (apply at next reboot).

7. Step 5 — Swarm tokens, remaining nodes

  1. node01 initialized the Swarm on first boot and printed join tokens. Read them: 
    journalctl -u swarm-join | grep -A2 'join-token'
  2. Store both SWMTKN-… values in 1Password: Infrastructure → Swarm → manager-token / worker-token.
  3. Repeat Step 4 (install → hardware config → token file → build) for node02…node10, using the matching name (#node02 etc.). Each node reads the right token for its role and joins by itself. Verify on any manager: docker node ls — all nodes Ready, 01–03 as managers.

8. Step 6 — Deploy Portainer

  1. On node01: cluster-deploy-portainer (deploys the Portainer server + an agent on every node, as Swarm services).
  2. From your workstation (on the tailnet), open https://node01:9443, create the admin account, and save the password in the Admin vault (item Portainer Admin).
  3. You should see all ten nodes with CPU/RAM/containers. Portainer is your window, not your source of truth — durable changes still go through Git.

9. Step 7 — Ship logs to Better Stack

  1. In telemetry.betterstack.com → Sources → Connect source, create a source (platform: Vector or HTTP). Copy its token and ingest URL.
  2. Token → 1Password: Infrastructure → BetterStack → credential.
  3. Ingest URL → ops repo cluster-settings.nix → logging.endpoint. Commit, push, then cluster-rebuild on each node.
  4. Every node now ships its full journal — including all container logs — via Vector. Set up alerts in Better Stack (e.g. on FAILED patterns or log-volume anomalies) and point them at e-mail now, webhooks later (that's the support agent's trigger).
Why Better Stack, and the exit ramp. Chosen for ease of setup: hosted, OpenTelemetry-native, free tier, built-in alerting/webhooks, and an MCP server AI agents can query. Because shipping goes through Vector (vendor-neutral), migrating later to self-hosted Loki or ClickStack is: stand up the backend → change logging.endpoint → swap the 1Password credential. Dashboards and alert rules are what you'd recreate.

11. Day-2 operations

  • Change anything → edit the ops repo, push, cluster-rebuild on affected nodes. Manual hotfixes on a node must be backported to the repo — an unmerged fix is debt.
  • Per-node software (GPU drivers, an accelerator card, a one-off service) → create hosts/nodeXX/extra.nix in the ops repo. It's auto-imported; shared modules stay untouched.
  • Roll back a bad change → nixos-rebuild switch --rollback on the node returns to the previous generation; then fix the repo.
  • Decommission a node → drain in Portainer, docker node rm, remove from Tailscale admin, revoke its 1Password service-account token if compromised.
  • Support agent (planned) → each node will run a static-binary AI agent (spec: github.com/SEBK4C/Nix-Support-Agent). You'll open an issue in the ops repo labeled node/node07 ("install ROCm"), and the node's agent diagnoses, implements via PR, and reports back. Better Stack alert webhooks trigger it automatically for incidents.
  • Keep current → periodically nix flake update in the ops repo (updates NixOS packages), test with cluster-rebuild test on one worker before rolling out.

12. Scaling to Dev / Staging / Production

The whole design is parameterized by vault name. To run multiple environments:

  1. Create one vault per environment: Infra-Dev, Infra-Staging, Infra-Prod — each containing the same item names from Section 3.2 (different credential values: separate tailnets or ACL-segmented tags, separate Better Stack sources, etc.).
  2. Create one service account per environment, each granted read access to exactly its own vault (nixos-cluster-dev → Infra-Dev only). Store each token in the Admin vault.
  3. In each environment's ops repo (or branch), set cluster.vault = "Infra-Dev"; in cluster-settings.nix. That single line selects the environment — every op:// reference follows it.
  4. Blast-radius rule: a compromised dev node can never read staging or production secrets, because its token simply has no grant there.
